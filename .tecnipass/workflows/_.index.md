# Workflows

> Reusable, end-to-end orchestration sequences that coordinate agents, protocols, and harness layers to complete a class of work.

## Contents

| File | Trigger | When to load |
|---|---|---|
| `init-session.workflow.md` | Any new task or session start | Always — this is the universal entry point for all work |
| `prompt-intake.workflow.md` | Substantial human prompt that starts, resumes, redirects, or validates work | Before task decomposition, to archive the original prompt and decide resume vs new session |
| `harness-evolution.workflow.md` | Long session, harness drift, repeated corrections, token inefficiency, or reusable harness pattern discovered | Only when there is evidence that the harness itself needs improvement |
| `harness-evolution.state.json` | Lightweight execution state for harness evolution | Read before loading `harness-evolution.workflow.md` |

## What Is a Workflow

A workflow is an ordered sequence of steps that references multiple harness layers (agents, protocols, tasks, sessions, memory, project, architecture, skills) to accomplish a category of work. Workflows do not duplicate layer content — they orchestrate it.

Think of it as:
- **Protocols** define rules (what must hold true).
- **Agents** define roles (who does what).
- **Workflows** define sequences (in what order, with what decisions).

## Access Pattern

1. The orchestrator (or the first agent entering the harness) reads `workflows/_.index.md`.
2. For substantial human prompts, run `prompt-intake.workflow.md` before task decomposition.
3. Based on the work category, load the matching `.workflow.md`.
4. For any long workflow, first run the lightweight Harness Evolution Gate:
   - read `workflows/harness-evolution.state.json`;
   - check whether a trigger has evidence and whether cooldown should be respected;
   - load `harness-evolution.workflow.md` only when the gate passes.
5. The selected workflow drives the session from init to close, referencing protocols and agents by filename.

## Harness Evolution Gate

Use this gate during long workflows, session close, checkpoint creation, or after repeated corrections.

Execute `harness-evolution.workflow.md` only when:
- there is concrete evidence of harness drift, regression, repeated correction, token waste, stale references, or useful reusable pattern;
- and either `cooldown_hours` has elapsed since `last_executed_at`, or the evidence matches `bypass_cooldown_when`.

Skip it when:
- the session is short;
- the concern is only aesthetic;
- the last execution is recent and no severe trigger exists;
- product work can continue safely without harness changes.

## Contributing

- New workflows use `{name}.workflow.md` naming.
- A workflow must reference existing protocols and agents — never redefine their rules inline.
- Keep workflows under 200 lines. If longer, split into phases as separate files and reference from a parent workflow.
- Every decision point must have explicit branches with clear criteria — no ambiguous "use judgment" steps.
- Workflows are provider-agnostic — they describe what happens, not which tool runs it.
- Long workflows must be aware of `harness-evolution.workflow.md`, but must not execute it without evidence because it consumes tokens.
- Prompt intake preserves the original user request; execution state belongs in `tasks/` and `sessions/`.

## Rules

- A workflow is a coordination document, not a specification or a protocol.
- Workflows reference layers by filename: `protocols/charter-preflight.protocol.md`, `agents/orchestrator.agent.md`, etc.
- Only the orchestrator (or the entry agent) loads workflows. Specialist agents receive context packs, not workflow files.
- When a workflow step says "load protocol X," the agent reads that protocol and follows it — the workflow does not paraphrase the protocol.
