# Architecture

> Software, infrastructure, cloud, and logical system architecture specifications for TecniPass.

## Contents

| File | Scope | When to load |
|---|---|---|
| `backend.architecture.md` | NestJS monorepo layout, DDD layers, file naming, technology guidelines | Backend feature work, API design, database migrations |
| `frontend.architecture.md` | Remix scaffolding, route conventions, component structure, CSS patterns | Frontend feature work, route creation, component design |
| `cloud-aws.architecture.md` | AWS deployment, multi-environment layout, VPC, databases, VPN connectivity | Infrastructure changes, deployment config, environment setup |
| `logical-services.architecture.md` | Microservice boundaries, Redis/BullMQ queues, device integrations, connectivity flows | Cross-service work, queue design, hardware integration |

## Access Pattern

1. Load only the architecture file matching your task domain.
2. For cross-cutting changes (e.g., new microservice touching frontend + backend + infra), load all relevant files.
3. Compare proposed changes against architecture rules before implementation.

## Contributing

- New architecture files use `{domain}.architecture.md` naming.
- Include frontmatter with `name` and `description`.
- Document the current state, not aspirational state — keep separate ADRs for planned changes.
- When implementation changes architecture (new service, new AWS resource, new integration pattern), update the corresponding file immediately.

## Rules

- Do not assume infrastructure features exist without verifying in these documents.
- Architecture files are the authority on system topology — code must conform to them.
- Changes to architecture require an ADR entry in `memory/decisions.md`.
