# Sessions

> Session logs, snapshots, and handoff records. A session is a bounded work period by one or more agents.

## Contents

| File | Purpose | When to load |
|---|---|---|
| `_template.session.md` | Scaffold for creating new session records | Starting a new session |
| `_snapshot.template.md` | Scaffold for session snapshots (state capture for resume) | Creating a checkpoint or handoff |

## Access Pattern

1. For substantial prompts, run `workflows/prompt-intake.workflow.md` and check `memory/prompts/` for similar prior work.
2. Check for active/recent snapshots before starting new work — a prior session may need resuming.
3. On session start, create a session file from `_template.session.md` and link `prompt_ref` when available.
4. At checkpoints or before context reset, create a snapshot from `_snapshot.template.md`.
5. During long sessions or repeated corrections, run the lightweight Harness Evolution Gate from `workflows/_.index.md`.
6. On session end, finalize the session file with: outcome, decisions made, memory updates, next steps.

## Contributing

- Session files use `{date}-{agent-or-task}-{description}.session.md` naming.
- Snapshots use `{date}-{description}.snapshot.md` naming.
- Include enough context in snapshots for a fresh agent to resume without reading the full session log.
- Reference (don't duplicate) specs, tasks, and decisions by filename.
- Reference prompt archives as `prompt_ref: memory/prompts/YYYY-MM-DD-{short-name}.prompt.md`.

## Naming Convention

```
2026-05-27-frontend-reception-offline.session.md
2026-05-27-reception-checkpoint.snapshot.md
```

## Rules

- Sessions are append-only during execution — no retroactive edits.
- A snapshot must be self-contained: an agent reading only the snapshot can resume work.
- Follow `protocols/session-lifecycle.protocol.md` for creation, checkpoint, and close procedures.
- Follow `protocols/handoff.protocol.md` when transferring context between agents or sessions.
- Before closing a long or drift-prone session, read `workflows/harness-evolution.state.json`; load `workflows/harness-evolution.workflow.md` only if evidence justifies the token cost.
- Completed sessions can be archived or pruned — they are not permanent memory (that goes in `memory/`).
