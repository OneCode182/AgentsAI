# Specs

> Feature and fix specifications that define what must be built. Each spec is the contract between product intent and agent execution.

## Contents

| File | Purpose | When to load |
|---|---|---|
| `_template.spec.md` | Scaffold for creating new specifications | Creating a new spec |
| `example-reception-feature.spec.md` | Reference example of a well-formed feature spec | Learning the spec format |
| `tanstack-specs/` | TanStack-specific form and query specifications | Frontend form or data-fetching tasks |

## Access Pattern

1. Agent receives a spec filename from the task or orchestrator.
2. Agent reads only the assigned spec — never browse all specs.
3. Spec defines: objective, acceptance criteria, affected files, constraints, and done definition.
4. Agent must run `protocols/charter-preflight.protocol.md` against the spec before implementation.

## Contributing

- Copy `_template.spec.md` to `{feature-or-fix}.spec.md`.
- Tie to a Linear ticket: include `TEC-XXX` in frontmatter.
- Keep specs under 100 lines — split complex features into sub-specs.
- Acceptance criteria must be testable and unambiguous.
- Reference business rules from `project/business-rules.md` by ID (e.g., D-01, R-03) instead of restating them.

## Naming Convention

```
{ticket-or-area}-{short-description}.spec.md
```

Examples: `tec-500-doorman-alert-config.spec.md`, `reception-offline-fallback.spec.md`.

## Rules

- A spec is frozen once execution begins — changes require a new version or amendment section.
- Specs reference `project/` for business rules and `architecture/` for technical constraints — they do not duplicate that content.
- Every spec must have a `## Done Definition` section that the QA agent can verify.
