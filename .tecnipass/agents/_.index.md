# Agents

> Role definitions for LLM agents operating within the TecniPass harness.

## Contents

| File | Role | When to load |
|---|---|---|
| `orchestrator.agent.md` | Task decomposition, execution mode selection, multi-agent coordination, quality gates | Starting any multi-step or multi-agent task |
| `frontend.agent.md` | Remix routes, React components, hooks, TanStack, Tailwind, accessibility | Frontend implementation tasks |
| `backend.agent.md` | NestJS modules, use cases, DTOs, persistence, RBAC, API contracts | Backend implementation tasks |
| `reviewer.agent.md` | Code review: bugs, regressions, security, architecture adherence | Code review or PR review tasks |
| `qa.agent.md` | Acceptance criteria validation, test strategy, regression analysis, lint evidence | QA validation or release assessment |
| `git-master.agent.md` | Atomic commits, branch comparison, rebase planning, conflict resolution | Git operations, commit preparation, history inspection |
| `_template.agent.md` | Scaffold for creating new agent definitions | Creating a new agent role |

## Access Pattern

1. Orchestrator reads its own definition first, then assigns tasks to specialist agents.
2. Specialist agents read only their own `.agent.md` — never other agents.
3. Agent file defines: role, owned domains, required skills to load, input/output contracts, and behavioral rules.

## Contributing

- Copy `_template.agent.md` to `{role}.agent.md`.
- Fill every section in the template — no optional fields.
- Keep frontmatter `description` under 25 words — it is used for routing.
- Register the new agent in root `AGENTS.md` work navigation table.

## Rules

- An agent must not act outside its declared domain.
- Agent files are read-only during execution — never self-modify.
- Agents load skills from `skills/` as declared in their definition, not ad-hoc.
