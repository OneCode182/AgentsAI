# Protocols

> Provider-agnostic workflow contracts that govern agent behavior, session lifecycle, and quality assurance.

## Contents

| File | Purpose | When to load |
|---|---|---|
| `charter-preflight.protocol.md` | Ambiguity, scope, and success-criteria check before any work begins | Always — first protocol loaded after boot |
| `execution-mode.protocol.md` | Choose sequential, parallel, or Agent Team execution based on task shape | After preflight, before implementation |
| `parallel-agent-architecture.md` | Multi-agent coordination: fork-join, shared task lists, specialist routing | When execution mode is `parallel` or `team` |
| `task-board.protocol.md` | Shared Task List schema and ownership rules across execution modes | Parallel or team execution |
| `context-budget.protocol.md` | Token budget, checkpoint, and reset rules for long sessions | Long-running or multi-step sessions |
| `context-loading.protocol.md` | Progressive context loading — what to load and when | Any session needing harness context |
| `quality-gate.protocol.md` | Verification criteria before marking work as done | Before completing any task |
| `verification-retry.protocol.md` | Retry and escalation loop for failed verifications | When quality gate fails |
| `decision-log.protocol.md` | Lightweight decision and progress logging | During any implementation session |
| `session-lifecycle.protocol.md` | Session creation, snapshot, resume, and close rules | Session management |
| `handoff.protocol.md` | Context transfer between sessions or agents | Resuming work or switching agents |
| `harness-engineering-principles.md` | Meta-rules for evolving the harness itself | Modifying harness structure or adding protocols |

## Access Pattern

1. Every session loads `charter-preflight` → `execution-mode` as mandatory boot sequence.
2. Remaining protocols are loaded on demand based on execution mode and task type.
3. Protocols reference each other by filename — follow references when encountered.

## Contributing

- New protocols use `{name}.protocol.md` naming.
- Include frontmatter with `name` and `description` fields.
- Keep protocols under 150 lines — split into sub-protocols if larger.
- A protocol defines **what** must happen, not **how** a specific tool does it.

## Rules

- Protocols are provider-agnostic — no references to specific LLM providers, IDEs, or CLI tools.
- Protocol compliance is mandatory — agents cannot skip protocols unless explicitly exempted by the orchestrator.
- When protocols conflict, authority order is defined in root `AGENTS.md` section 3.
