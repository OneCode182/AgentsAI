# Tasks

> Active work items, execution checklists, and task boards. Tasks are the runtime state of specs being executed.

## Contents

| File / Dir | Purpose | When to load |
|---|---|---|
| `_template.task.md` | Scaffold for creating new task files | Creating a new task |
| `active/` | Directory for currently in-progress task files | Resuming or checking active work |

## Access Pattern

1. Orchestrator creates tasks from specs using `_template.task.md`.
2. Each task references its parent spec, prompt archive when available, and assigned agent.
3. Active tasks live in `active/` — completed tasks are archived or deleted.
4. For multi-agent execution, the shared task board lives in `active/` and follows `protocols/task-board.protocol.md` schema.

## Contributing

- Copy `_template.task.md` to `active/{ticket-or-area}-{description}.task.md`.
- Task must reference its parent spec file.
- Task should reference `prompt_ref` when created from an archived prompt.
- Update task status as work progresses: `pending` → `in-progress` → `blocked` → `done`.
- On completion, move task out of `active/` or mark status as `done`.

## Naming Convention

```
active/{ticket}-{short-description}.task.md
```

Examples: `active/tec-500-implement-alert-api.task.md`, `active/reception-fix-qr-scan.task.md`.

## Rules

- Tasks are ephemeral — they represent current work, not historical record.
- A task without a parent spec is valid only for bug fixes or operational chores.
- Task status is the source of truth for "what is being worked on right now."
- When a task reveals a new decision or mistake, append to `memory/decisions.md` or `memory/mistakes.md`.
