## Access Pattern

1. Load only the file matching the task's product domain.
2. For cross-domain tasks, load the relevant files plus `business-rules.md`.
3. `business-rules.md` is the single reference when checking constraints — rules there are indexed by ID (D-01, R-01, G-01) for traceability.

## Contributing

- Epic-level only — no implementation details, no code references.
- Each epic references its Linear ticket as `TEC-XXX`.
- Business rules tagged: `[DEFINED]` (absolute invariant), `[RELATIVE]` (context-dependent), `[GENERAL]` (cross-cutting convention).
- Acceptance criteria use imperative form: "System must...", "User can...".
- When a new epic is created in Linear, add it to the matching emphasis-line file.
- When a new business rule is discovered, add it to both the emphasis-line file and `business-rules.md`.

## Rules

- This layer describes **what** the product does, not **how** it is built.
- Do not invent business rules — verify against Linear tickets and stakeholder specs.
- Cross-reference `../project.md` for technical stack context; this layer is product-only.
